# Руководство по эксплуатации и развертыванию

## Область применения
- Первое использование на рабочем столе
- Service Версия работает независимо
- Docker Развертывание
- macOS Первый выпуск

## Быстрый старт
1. Запустите рабочий стол и нажмите «Запустить службу».
2. Войдите в «Управление учетной записью», добавьте учетную запись и завершите авторизацию.
3. Если обратный вызов завершился неудачно, вставьте ссылку обратного вызова, чтобы завершить анализ вручную.
4. Обновите использование и подтвердите статус учетной записи.

## Подключение через ccswitch
Если вы используете CodexManager через ccswitch или напрямую из Codex CLI, синхронизируйте platform key, `auth.json` и `config.toml`.

1. Откройте `Platform Keys` и создайте универсальный key.
2. В ccswitch создайте provider и вставьте этот key в поле API Key.
3. Запишите тот же key в `auth.json` Codex CLI. Не используйте здесь `access_token`, `refresh_token` аккаунта или OpenAI login token.
4. Скопируйте пример `config.toml` ниже в конфиг ccswitch / Codex и перезапустите Codex CLI.

Типичные пути:

- macOS / Linux: `~/.codex/auth.json`, `~/.codex/config.toml`
- Windows: `%USERPROFILE%\.codex\auth.json`, `%USERPROFILE%\.codex\config.toml`

Пример `auth.json`:

```json
{
  "OPENAI_API_KEY": "replace_with_codexmanager_platform_key",
  "auth_mode": "apikey"
}
```

Пример `config.toml`:

```toml
model = "gpt-5.4"
model_provider = "cm"
review_model = "gpt-5.4"
approval_policy = "on-request"
sandbox_mode = "workspace-write"
cli_auth_credentials_store = "file"
service_tier = "fast"

[model_providers.cm]
name = "OpenAI"
base_url = "http://localhost:48760/v1"
wire_api = "responses"
supports_websockets = true
```

- Если порт service изменён в Settings, обновите `base_url`.
- После изменения `auth.json` или `config.toml` перезапустите Codex CLI.

## Импорт и экспорт аккаунтов
- `批量导入`: выберите несколько файлов `.json/.txt` и импортируйте их вместе.
- `按文件夹导入`: доступно только на рабочем столе; после выбора каталога рекурсивно сканируйте файлы `.json` и импортируйте их пакетами. Пустые файлы будут автоматически пропущены.
- `导出用户`: после выбора каталога нажмите «Один файл JSON для каждой учетной записи» для экспорта, чтобы облегчить резервное копирование и миграцию.

## Service Версия
1. Загрузите `CodexManager-service-<platform>.zip` в Release и разархивируйте его.
2. Рекомендуется начать с `codexmanager-start`. Процесс запускает сервис + веб и может быть закрыт прямо в консоли `Ctrl+C`.
3. Вы также можете просто запустить `codexmanager-web`, который автоматически откроет `codexmanager-service` в том же каталоге и откроет браузер.
4. Или начните сначала `codexmanager-service`, а затем `codexmanager-web`.
5. Адрес по умолчанию: сервис `localhost:48760`, Web UI `http://localhost:48761/`.
6. Закрытие: посетите `http://localhost:48761/__quit`; если веб-сервис был запущен автоматически, он попытается закрыть его вместе.
7. Если вам необходимо выполнить обратный прокси-сервер самостоятельно или разделить статические ресурсы внешнего интерфейса, вы должны перенаправить `/api/runtime` и `/api/rpc` одновременно; только размещение статических файлов приведет к неправильной работе страницы управления.

## Docker Развертывание

### GitHub Пакеты / GHCR
- После выпуска Release образы `codexmanager-service` и `codexmanager-web` будут одновременно перенесены в пакеты GitHub (GHCR).
- Просто потяните соответствующий тег выпуска, например: `docker pull ghcr.io/qxcnm/codexmanager-service:v0.1.15`.
- [`docker/docker-compose.release.yml`](../../../docker/docker-compose.release.yml) на складе также напрямую ссылается на GHCR, установите `CODEXMANAGER_RELEASE_TAG` перед использованием.
- Пример: `CODEXMANAGER_RELEASE_TAG=v0.1.15 docker compose -f docker/docker-compose.release.yml up -d`

### Способ 1: создание докера
``` баш
docker compose -f docker/docker-compose.yml up --build
```

Браузер открывается: `http://localhost:48761/`

### Способ 2: сборка и запуск отдельно
``` баш
#сервис
docker build -f docker/Dockerfile.service -t codexmanager-service .
docker network create codexmanager-net

docker run --rm --name codexmanager-service \
  --network codexmanager-net \
  --network-alias codexmanager-service \
  -p 48760:48760 \
  -v codexmanager-data:/data \
  -e CODEXMANAGER_RPC_TOKEN=заменить_на_ваш_токен \
  codexmanager-сервис

# веб (между контейнерами используйте адрес Docker-сети, а не порт хоста)
docker build -f docker/Dockerfile.web -t codexmanager-web .
docker run --rm --name codexmanager-web \
  --network codexmanager-net \
  -p 48761:48761 \
  -v codexmanager-data:/data \
  -e CODEXMANAGER_WEB_NO_SPAWN_SERVICE=1 \
  -e CODEXMANAGER_SERVICE_ADDR=codexmanager-service:48760 \
  -e CODEXMANAGER_RPC_TOKEN=заменить_на_ваш_токен \
  codexmanager-web
```

- Примечание. Если вы хотите, чтобы пароль Web, элементы настройки, список моделей кэша и другой статус соответствовали службе, `codexmanager-web` и `codexmanager-service` должны использовать один и тот же том `/data`.
- Если контейнеру всё же нужно ходить в порт хоста, на Linux добавьте `--add-host=host.docker.internal:host-gateway`; без этого `host.docker.internal` часто не резолвится.

## macOS Первый запуск
- Текущая версия продукта macOS не была нотариально заверена с использованием учетной записи разработчика Apple, поэтому после первой загрузки из браузера Gatekeeper может выдать сообщение «Поврежден» или отказаться открывать.
- macOS `dmg` в версии Release имеют встроенные `Open CodexManager.command` и `README-macOS-first-launch.txt`.
- Рекомендуется сначала перетащить `CodexManager.app` в «Приложения», а затем дважды щелкнуть сценарий, чтобы завершить первый выпуск.
- Вы также можете выполнить его напрямую:

``` баш
xattr -dr com.apple.quarantine /Applications/CodexManager.app
```

- Если он по-прежнему заблокирован, снова выполните «щелчок правой кнопкой мыши -> открыть» на `CodexManager.app`.
