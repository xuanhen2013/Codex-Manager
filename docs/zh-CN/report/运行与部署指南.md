# 运行与部署指南

## 适用范围
- 桌面端首次使用
- Service 版本独立运行
- Docker 部署
- macOS 首次放行

## 快速开始
1. 启动桌面端，点击“启动服务”。
2. 进入“账号管理”，添加账号并完成授权。
3. 如回调失败，粘贴回调链接手动完成解析。
4. 刷新用量并确认账号状态。

## 通过 ccswitch 接入
如果你希望通过 ccswitch 使用 CodexManager，可按下面步骤操作：

1. 进入“平台密钥”，新建一个通用 Key；如需让 ccswitch / Codex CLI 长期复用固定 `OPENAI_API_KEY`，可在创建时填写“自定义 API Key”，否则留空自动生成。
2. 进入 ccswitch，新建一个供应商，并把上一步生成或自定义的通用 Key 填入供应商里的 API Key 空白处。
3. 将同一个通用 Key 写入 Codex CLI 使用的 `auth.json`。这里不要填写账号里的 `access_token`、`refresh_token` 或 OpenAI 登录 token。
4. 将下面示例 `config.toml` 复制到 ccswitch / Codex 使用的 `config.toml` 中，保存后重启 Codex CLI。

常见路径：
- macOS / Linux：`~/.codex/auth.json`、`~/.codex/config.toml`
- Windows：`%USERPROFILE%\.codex\auth.json`、`%USERPROFILE%\.codex\config.toml`

示例 `auth.json`：

```json
{
  "OPENAI_API_KEY": "填入你在 CodexManager 平台密钥里复制的 Key",
  "auth_mode": "apikey"
}
```

补充说明：

- `auth.json` 和 `config.toml` 应放在同一个 `.codex` 目录下。
- 如果 `auth.json` 已经存在，只需要替换其中的 `OPENAI_API_KEY`；使用自定义平台 Key 时，这里应填写同一个自定义值。
- 修改 `auth.json` 后请重新启动 Codex CLI，避免旧认证缓存继续生效。

示例 `config.toml`：

```toml
model = "gpt-5.4"
model_provider = "cm"
review_model = "gpt-5.4"
personality = "none"
model_reasoning_effort = "xhigh"
plan_mode_reasoning_effort = "xhigh"
model_reasoning_summary = "detailed"
model_verbosity = "medium"
model_supports_reasoning_summaries = true
approval_policy = "on-request"
allow_login_shell = true
sandbox_mode = "workspace-write"
cli_auth_credentials_store = "file"
chatgpt_base_url = "https://chatgpt.com/backend-api/"
mcp_oauth_credentials_store = "auto"
check_for_update_on_startup = true
web_search = "live"
approvals_reviewer = "user"
service_tier = "fast"

[model_providers.cm]
approval_policy = "on-request"
sandbox_mode = "workspace-write"
web_search = "live"
name = "OpenAI"
base_url = "http://localhost:48760/v1"
wire_api = "responses"
```

- 如果你在设置页改过服务端口，请把 `base_url` 里的 `48760` 同步改成当前端口。
- Docker / Web 网关模式下，ccswitch / Codex CLI 应连接浏览器可访问的 Web 地址并追加 `/v1`。例如 Web UI 是 `http://10.10.121.253:48761/` 时，`base_url` 应写成 `http://10.10.121.253:48761/v1`，不要写容器内的 `localhost:48760`。

## 账号导入与导出
- `批量导入`：选择多个 `.json/.txt` 文件后统一导入。
- `按文件夹导入`：仅桌面端可用；选择目录后递归扫描其中 `.json` 文件并批量导入，空文件会自动跳过。
- `导出用户`：选择目录后按“一个账号一个 JSON 文件”导出，便于备份与迁移。

## Service 版本
1. 下载 Release 中的 `CodexManager-service-<platform>-<arch>.zip` 并解压。
2. 推荐启动 `codexmanager-start`，一个进程拉起 service + web，可直接在控制台 `Ctrl+C` 关闭。
3. 也可以只启动 `codexmanager-web`，它会自动拉起同目录的 `codexmanager-service` 并打开浏览器。
4. 或者先启动 `codexmanager-service`，再启动 `codexmanager-web`。
5. 默认地址：service `localhost:48760`，Web UI `http://localhost:48761/`。
6. 关闭：访问 `http://localhost:48761/__quit`；若 web 自动拉起过 service，会尝试一并关闭。
7. 如需自行反向代理或拆分部署前端静态资源，必须同时转发 `/api/runtime` 与 `/api/rpc`；仅托管静态文件会导致管理页面无法正常工作。

### 从源码运行内嵌前端版本

源码运行时推荐使用：

```bash
./scripts/run-service-app.sh
```

脚本会先执行 `pnpm -C apps run build:desktop` 生成 `apps/out`，再编译 `codexmanager-service`、`codexmanager-web` 和 `codexmanager-start`。`codexmanager-web` 默认会把 `apps/out` 打进二进制，因此启动时不需要再手动设置 `CODEXMANAGER_WEB_ROOT`。

常用参数：

- `--debug`：使用 debug 构建，加快本地调试编译。
- `--clean-dist`：构建前清理 `apps/out`。
- `--no-open`：启动后不自动打开浏览器。

如果需要临时使用外置静态资源目录，仍可设置 `CODEXMANAGER_WEB_ROOT=/path/to/out` 覆盖内嵌资源。

## Docker 部署

Docker 镜像默认设置 `TZ=Asia/Shanghai`，compose 示例使用 `TZ=${TZ:-Asia/Shanghai}`：如果部署环境已经设置 `TZ`，会直接沿用；否则回退到 `Asia/Shanghai`。如果部署在其他地区，请设置 `TZ` 或在 compose 的 `environment` 中改成自己的 IANA 时区，例如 `Europe/London`、`America/Los_Angeles`。

### GitHub Packages / GHCR
- Release 发布后会同时推送 `codexmanager-service` 和 `codexmanager-web` 镜像到 GitHub Packages（GHCR）。
- 直接拉取对应发布 tag 即可，例如：`docker pull ghcr.io/qxcnm/codexmanager-service:v0.1.15`
- 仓库里的 [`docker/docker-compose.release.yml`](../../../docker/docker-compose.release.yml) 也直接引用 GHCR，使用前先设置 `CODEXMANAGER_RELEASE_TAG`。
- 例如：`CODEXMANAGER_RELEASE_TAG=v0.1.15 docker compose -f docker/docker-compose.release.yml up -d`

### 方式 1：all-in-one 单容器
```bash
docker compose -f docker/docker-compose.all-in-one.yml up -d --build
```

浏览器打开：`http://localhost:48761/`

- 该模式只构建一个镜像，容器内由 `codexmanager-start` 同时拉起 `codexmanager-service` 和 `codexmanager-web`。
- 默认只暴露 `48761`。Web 管理页面、`/api/rpc` 和 OpenAI 兼容网关路径（如 `/v1/responses`、`/v1/chat/completions`、`/v1/messages`）都从该端口进入；`48760` 仅在容器内部由 web 反代到 service。
- 远程服务器上使用 ccswitch 导入时，endpoint 应使用当前 Web UI 地址生成，例如 `http://10.10.121.253:48761/v1`。如果显示成 `localhost`，请先手动改为服务器地址加 `/v1`。
- 数据库、RPC token、Web 密码和运行设置都保存在同一个 `/data` 卷里。

### 方式 2：docker compose 双容器
```bash
docker compose -f docker/docker-compose.yml up --build
```

浏览器打开：`http://localhost:48761/`

### 方式 3：分别构建和运行
```bash
# service
docker build -f docker/Dockerfile.service -t codexmanager-service .
docker network create codexmanager-net

docker run --rm --name codexmanager-service \
  --network codexmanager-net \
  --network-alias codexmanager-service \
  -p 48760:48760 \
  -v codexmanager-data:/data \
  -e TZ=Asia/Shanghai \
  -e CODEXMANAGER_RPC_TOKEN=replace_with_your_token \
  codexmanager-service

# web（容器之间请走容器网络地址，不要写宿主机映射端口）
docker build -f docker/Dockerfile.web -t codexmanager-web .
docker run --rm --name codexmanager-web \
  --network codexmanager-net \
  -p 48761:48761 \
  -v codexmanager-data:/data \
  -e TZ=Asia/Shanghai \
  -e CODEXMANAGER_WEB_NO_SPAWN_SERVICE=1 \
  -e CODEXMANAGER_SERVICE_ADDR=codexmanager-service:48760 \
  -e CODEXMANAGER_RPC_TOKEN=replace_with_your_token \
  codexmanager-web
```

- 注意：若希望 Web 密码、设置项、缓存模型列表等状态与 service 保持一致，`codexmanager-web` 必须和 `codexmanager-service` 共享同一份 `/data` 卷。
- 若你坚持让容器访问宿主机映射端口，在 Linux 上还需要额外增加 `--add-host=host.docker.internal:host-gateway`；否则 `host.docker.internal` 往往无法解析。

## macOS 首次启动
- 当前 macOS Release 产物未使用 Apple Developer 账号完成公证，因此首次从浏览器下载后，Gatekeeper 可能提示“已损坏”或拒绝打开。
- Release 中的 macOS `dmg` 已内置 `Open CodexManager.command` 与 `README-macOS-first-launch.txt`。
- 推荐先把 `CodexManager.app` 拖到“应用程序”，再双击该脚本完成首次放行。
- 也可以直接执行：

```bash
xattr -dr com.apple.quarantine /Applications/CodexManager.app
```

- 如果仍被拦截，再对 `CodexManager.app` 执行一次“右键 -> 打开”。
