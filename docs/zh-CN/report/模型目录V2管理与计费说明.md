# 模型目录 V2 管理与计费说明

本文说明当前模型目录、路由、instructions policy、缓存导出和钱包计费的实际行为。模型目录 V2 已一次性接管运行时；旧模型来源、旧价格规则和供应商模型发现表只为迁移审计保留，不再参与请求。

## 1. 唯一真相源

- 模型来源只有 `builtin` 和 `custom`。
- fresh DB 会写入 8 个 builtin：7 个 `visibility=list` 模型立即显示在普通列表中，`codex-auto-review` 为 `visibility=hide`，只能通过 hidden 筛选管理或按 slug 显式请求。
- 本地 `/v1/models` 只读取 enabled、`visibility=list`、`supportedInApi=true` 的 V2 数据，不联网、不补建记录、不写数据库。
- 启动、模型页、账号切换、聚合 API 页面和 Gateway 都不会请求上游或供应商 `/models`。
- 其他模型通过“新增自定义模型”手工添加，或从本地 JSON 导入。

`gpt-5.6-sol`、`gpt-5.6-terra`、`gpt-5.6-luna` 已分别写入 5/30、2.5/15、1/6 USD / 1M tokens 的 input/output 价格。因当前来源未单列 cached input，缓存输入暂按普通 input 同价保守计费，状态标为 `estimated`；hidden 的 `codex-auto-review` 仍明确显示 `price missing`，不是零价。

## 2. 模型管理页

列表同时展示：

- origin：`builtin` / `custom`
- enabled 与 visibility
- price status：`official` / `estimated` / `custom` / `missing`
- instructions mode：`passthrough` / `fallback` / `override`
- enabled route 数量或 `missing route`

统计卡片显示 total、enabled、builtin、custom、price missing 和 missing route。hidden 模型不会混入默认列表，选择 hidden 筛选后才会出现。

删除语义：

- builtin 只能禁用，记录、价格和审计信息不会被物理删除。
- custom 可以物理删除。
- 删除聚合 API 连接时，会删除引用该连接的 V2 routes；不会反写 legacy 模型表。

## 3. 一次原子保存

模型编辑器分为四区，点击一次“保存模型”只调用一次 V2 upsert RPC，并由一个数据库事务保存：

1. 基本信息和关键 capabilities。
2. input、cached input、output 三价，以及可选 long-context tier。
3. `account_pool` / `aggregate_api` routes；上游模型名手工填写。
4. instructions mode 和可选文本。

同一事务还会保存 permission group 关联。任一步校验或写入失败，整次保存回滚，不会出现“模型保存成功但价格或 route 保存失败”的半状态。

价格 UI 使用 USD / 1M tokens；提交时从十进制字符串无损转换为整数 micro-USD。最多支持 6 位有效小数，禁止负数、非数字、部分三价和超出 JavaScript 安全整数范围的值。

长上下文 tier 的 `minInputTokens` 是包含 cached tokens 的总 input token 阈值。系统选择不大于实际 input tokens 的最高档；例如当前 `gpt-5.4` / `gpt-5.5` 在 272000 tokens 切换到长上下文价格。

## 4. 路由与聚合 API

每条 V2 route 包含：

- `sourceKind`：`account_pool` 或 `aggregate_api`
- `sourceId`：账号池使用 `default`，聚合来源使用连接 ID
- `upstreamModel`
- enabled、priority、weight

builtin 默认拥有 `account_pool/default` route，上游模型等于平台 slug。aggregate route 只能由用户在模型编辑器中明确保存。

聚合 API 页面只管理连接、启停、密钥、余额和 route 连通性测试。测试会从引用该连接的 enabled V2 routes 中选择具体上游模型，不会发现、同步或导入供应商模型，也不会调用供应商 `/models`。

Gateway 收到平台 slug 后按以下顺序处理：

1. 读取 enabled V2 model，并检查 `supportedInApi`、权限组和计费条件。
2. 只读 enabled V2 routes，构造账号池和聚合 API 候选。
3. 为每个候选从不可变客户端 body 独立生成请求体，避免不同 provider 候选互相污染。
4. 将 route 的 `upstreamModel` 写入该候选请求。

## 5. Instructions policy

| mode | 行为 |
| --- | --- |
| `passthrough` | 客户端已有 instructions/system/developer 内容时原样保留；模型文本不参与请求。 |
| `fallback` | 任一客户端 instruction channel 非空时原样保留；全部为空时才使用模型文本。 |
| `override` | 使用非空模型文本替换顶层和连续 leading system/developer instructions。 |

builtin 默认是 `passthrough`，且 `instructions_text` 为空。seed 和 Codex 缓存导出都不携带官方人格 prompt。

真实 Codex target 最终仍执行 transport 级中性兜底：当最终 instructions 为 missing、null、空串或纯空白时，发送 `Follow the user's instructions.`。该文本不写入模型目录；非 Codex target 不接收此兜底。客户端非空内容保持原字符串，不 trim。

## 6. 本地 JSON 导入

“从本地 JSON 导入”支持：

- 模型目录 V2 导出结构。
- Codex catalog 的 `models` 结构。

流程固定为 preview 后 commit。preview 显示新增、更新、冲突、跳过、错误和忽略字段；冲突策略只有：

- `keep_existing`
- `replace_custom`

所有导入项都作为 custom 处理，不能覆盖 builtin。`base_instructions`、`instructions_template`、instructions variables 和未知 prompt 字段会被忽略；导入不联网。

## 7. Codex 缓存导出

桌面端和 Web 端都只在用户点击“导出到本地 Codex 缓存”后执行：

- 桌面端主动写入 `~/.codex/models_cache.json`。
- Web 端下载同名文件，由用户手工放入本地 `.codex` 目录。

保存、删除、读取、启动和后台 effect 不会自动写缓存。导出只包含 enabled、`supportedInApi=true`、`visibility=list` 的模型，并强制输出 `base_instructions: ""`。

GPT-5.6 的导出同时保留推理等级和 Codex 运行元数据：Sol、Terra 包含 `low` 到 `ultra` 以及 `multi_agent_version=v2`；Luna 包含到 `max` 且为 `multi_agent_version=v1`。导出还包含 `tool_mode`、`use_responses_lite`、`max_context_window`、`comp_hash`、verbosity、apply patch、truncation 和 speed tier 等非提示词字段，不输出 `model_messages`，也不会启用 `include_skills_usage_instructions`。

Max 与 Ultra 不是独立模型。Max 是所选模型的单代理最大推理强度；Ultra 由 Codex 客户端在 Max 推理强度之上主动编排子代理。兼容客户端如果直接向 Gateway 提交 `ultra`，请求日志分别保存 `client_reasoning_effort=ultra` 和最终 `reasoning_effort=max`，列表显示为 `ultra→max`。未修改的官方 Codex 会在到达 Gateway 前把 Ultra 的单次上游请求归一化为 Max，因此 Gateway 不能仅凭请求体可靠反推用户是否选择了 Ultra；`x-openai-subagent`、父线程和 turn metadata 仍会原样透传，但不会被错误推断为 Ultra。

## 8. 整数计费与快照

计费只使用 V2 DB 中实际命中的价格 tier：

```text
uncached_input = max(input_tokens - cached_input_tokens, 0)

numerator =
    uncached_input * input_rate
  + cached_input_tokens * cached_rate
  + output_tokens * output_rate

charged_cost_microusd =
  ceil(numerator * multiplier_millis / 1_000_000_000)
```

中间值使用 `i128`，倍率应用后只做一次最终向上取整。实际 usage 优先；缺 usage 或流式中断时使用确定性本地估算并标记 `estimated`，不会默认为 0。

价格读取、request charge snapshot、幂等 wallet ledger 和余额更新在一个 Core 事务中完成。快照保存实际 tier/rates、tokens、usage source、倍率、base cost 和 charged cost；请求 ID / request log ID 用作幂等键。后续修改价格不会改变历史快照和钱包流水。

`price_status=missing` 的模型可以在非钱包模式使用，也可以继续编辑和配置 route；它不能加入钱包扣费权限组。钱包请求会返回 `model_price_missing`，不会按零价放行，也不会 fallback 到编译期价格。

## 9. 迁移行为

首次 V2 cutover 会在维护态下排空在途 Gateway 流量和钱包写入，获取数据库旁跨进程锁，并使用 SQLite online backup 执行 preflight、单事务迁移和 smoke test。失败时在恢复流量前还原 backup。

legacy DB 的迁移规则：

- seed slug 复用对应 builtin 行；用户编辑字段、禁用状态和已验证显式价格优先。
- `114_model_catalog_gpt56_prices` 只为仍是 `missing` 且从未编辑的三款 builtin GPT-5.6 补价；任何自定义或已编辑价格都不会被覆盖。
- `115_model_catalog_codex_metadata` 把 revision 3 的非提示词 Codex 运行元数据补到未编辑 builtin；`user_edited=1` 的模型保持用户值。
- 旧 custom 与 `user_edited=1` 的非 seed 模型迁为 custom。
- 普通、未编辑 remote rows 不进入 V2。
- 重复执行 seed/migration 不覆盖用户编辑。
- legacy 表保留用于审计，但运行时不再读取或写入这些目录、来源、映射和价格规则。

## 10. 常见排查

### 模型不在普通列表

检查 enabled、visibility 和 `supportedInApi`。hidden 模型只在 hidden 筛选中出现；本地 `/v1/models` 也不会列出 hidden。

### 模型显示 missing route

在模型编辑器的“路由”页签添加 enabled route，并确认 aggregate route 的 source ID 对应仍存在且 active 的聚合 API。

### 钱包请求返回 `model_price_missing`

补齐 input/cached/output 三价后保存，或不要把该模型加入钱包扣费权限组。不要把空价格填写为 0 来绕过校验。

### 聚合 API 连通性测试提示没有 route

先在模型目录 V2 中为该连接保存一条 enabled aggregate route。聚合 API 页面不会自动发现上游模型。

### Codex `/model` 没有立即更新

主动导出缓存，并重启需要重新读取 `models_cache.json` 的 Codex 会话。模型保存本身不会自动改写缓存。
