# 账号模式与额度分发锁定策略

> 日期：2026-05-13
> 状态：设计决策，待实现约束
> 范围：Web 登录模式、额度分发、平台 Key 归属、成员模型组与钱包扣费

## 1. 背景

CodexManager 当前已经具备多用户账号、平台 Key 归属、成员钱包、模型组和请求扣费能力。随着这些能力组合使用，`Web 登录模式` 和 `额度分发` 不再只是普通偏好设置，而是会改变系统权限边界和账务语义的基础模式。

两个关键事实：

- 非账号系统模式下，Web RPC 没有明确登录用户身份，会按系统管理员语义处理。
- 额度分发开启后，平台 Key 会绑定归属钱包，请求会做钱包预检并在成功后扣费。

因此，一旦系统进入正式多用户计费状态，不应允许管理员在普通 UI 中随意回退到无账号、共享密码或关闭额度分发。

## 2. 目标

- 明确账号模式与额度分发的状态转换规则。
- 避免权限绕过：不能通过切换登录模式绕过成员权限与审计边界。
- 避免账务断层：不能通过关闭额度分发让请求绕过钱包扣费。
- 保留受控恢复手段：异常维护时允许本机、带备份和审计的恢复路径。
- 不引入项目维度。平台 Key 归属和额度消费只按用户钱包与模型组推进。

## 3. 核心结论

### 3.1 账号系统一旦正式启用，不允许从 UI 降级

账号模式进入正式使用后，不允许在普通设置页切回：

- `none`：不启用登录
- `password`：共享访问密码

原因：

- `none` 会让 Web 管理页公开访问。
- `password` 只能提供共享门禁，不能表达成员身份。
- 没有成员身份时，RPC 权限边界会退回系统管理员语义。
- 已经存在成员、平台 Key、钱包或请求日志后，回退会破坏后续权限审计。

### 3.2 额度分发一旦正式启用，不允许从 UI 关闭

额度分发进入正式使用后，不允许在普通设置页关闭。

原因：

- 关闭后不再做 Key 归属钱包预检。
- 关闭后请求成功不会扣成员钱包。
- 关闭期间产生的请求不会自动补账。
- 已配置的成员余额、模型组倍率、Key 归属会暂时失去约束，容易造成计费口径不一致。

### 3.3 额度分发必须依赖账号系统

启用额度分发前必须满足：

- Web 登录模式为 `accounts`。
- 至少有一个有效管理员账号。
- 平台 Key 只能归属到有效成员用户。

不允许在 `none` 或 `password` 模式下启用额度分发。

## 4. 正式使用信号

系统满足任一条件时，视为已经进入正式账号计费状态：

| 信号 | 说明 |
| --- | --- |
| 存在成员账号 | 除管理员外已有普通成员 |
| 存在平台 Key 归属 | 任一 Key 已绑定归属用户 |
| 存在成员钱包余额 | 任一成员钱包有余额或冻结余额 |
| 存在钱包流水 | 已发生充值、初始赠送或请求扣费 |
| 存在模型组成员分配 | 任一成员被分配到模型组 |
| 存在请求扣费记录 | 请求日志关联到钱包扣费流水 |

正式使用信号出现后，相关模式进入锁定状态。

## 5. 状态转换规则

### 5.1 登录模式

| 当前状态 | 目标状态 | 允许条件 | 说明 |
| --- | --- | --- | --- |
| `none` | `password` | 未锁定，且已设置访问密码 | 个人或可信小团队场景 |
| `none` | `accounts` | 允许 | 进入账号系统 |
| `password` | `none` | 未锁定 | 仅适合试用期回退 |
| `password` | `accounts` | 允许 | 进入账号系统 |
| `accounts` | `none` | 未锁定时允许；锁定后禁止 | 锁定后会造成权限降级 |
| `accounts` | `password` | 未锁定时允许；锁定后禁止 | 锁定后会丢失成员身份边界 |

### 5.2 额度分发

| 当前状态 | 目标状态 | 允许条件 | 说明 |
| --- | --- | --- | --- |
| 关闭 | 开启 | 登录模式必须为 `accounts` | 开启后 Key 需要归属钱包 |
| 开启 | 关闭 | 未锁定时允许；锁定后禁止 | 锁定后关闭会造成账务断层 |

## 6. UI 表达

普通设置页不应显示危险回退按钮。建议表现为：

- 未锁定时：允许切换，并显示风险提示。
- 已锁定时：禁用相关控件，只显示当前模式和锁定原因。
- 额度分发未开启且登录模式不是 `accounts` 时：禁用额度分发开关，并提示“请先启用账号系统”。

推荐文案：

```text
已进入账号计费模式。为避免权限绕过和账务断层，不能从界面关闭账号系统或额度分发。
```

锁定原因应尽量具体，例如：

```text
已存在平台 Key 归属和钱包流水，不能关闭额度分发。
```

## 7. 维护与恢复

锁定不应设计成无法恢复的死锁。异常情况下允许维护人员通过受控路径恢复：

- 只能在本机或受信任运维环境执行。
- 执行前必须备份数据库。
- 不在普通 Web 设置页暴露。
- 操作必须写入审计日志或维护记录。
- 恢复脚本应明确列出影响：登录模式、额度分发、Key 归属、钱包扣费、模型组权限。

维护入口可以是命令行、一次性迁移脚本或明确标记为危险操作的管理工具。

## 8. 与现有功能的关系

### 8.1 平台 Key

额度分发开启后，平台 Key 必须归属到有效成员用户。未归属、归属用户禁用、钱包停用或余额不足时，请求应被拦截。

### 8.2 模型组

模型组用于表达成员可调用的平台模型集合和计费倍率。成员可持有多个模型组，最终权限由模型组集合决定。

管理员账号不参与额度分发，不应被分配成员钱包。

### 8.3 计费规则

模型组倍率表达订阅层计费差异；模型基础价和长上下文阶梯价只读取模型目录 V2 的整数价格。钱包请求缺少模型价格时返回 `model_price_missing`，不会按零价放行，也不会回退到编译期或旧价格规则。

每个终态请求都会保存实际命中的价格阶梯、input/cached/output token、倍率、usage 来源和最终费用快照。钱包流水、余额更新和快照在同一事务中完成，后续改价不会改变历史扣费。

### 8.4 项目维度

项目维度不纳入本策略。后续实现应避免继续暴露项目归属入口，防止绕开成员模型组与用户钱包语义。

## 9. 实施建议

后续实现时建议分三步：

1. 增加锁定状态计算：由成员、Key 归属、钱包、模型组和流水判断是否已正式启用。
2. 后端限制危险切换：即使前端绕过，也不能通过 RPC 关闭已锁定模式。
3. 前端展示锁定原因：设置页只展示安全可执行的切换项和明确原因。

相关 RPC 应返回结构化错误，例如：

```text
account_billing_mode_locked
distribution_mode_locked
distribution_requires_accounts_mode
```

## 10. 验收口径

- 未正式使用时，可以在本地试用期间切换登录模式和额度分发。
- 启用账号系统并出现正式使用信号后，普通 UI 不能降级登录模式。
- 启用额度分发并出现正式使用信号后，普通 UI 不能关闭额度分发。
- 额度分发不能在 `none` 或 `password` 模式下开启。
- 被锁定的切换即使直接调用 RPC 也会失败。
- 维护恢复路径不出现在普通设置页。
