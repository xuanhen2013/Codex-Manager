# Codex 上游中转兼容跟踪清单 - 2026-06-09

## 1. 结论

CodexManager 主要是中转网关、账号池和管理后台，不需要实现上游 Codex 的 `app-server`、TUI、MCP、sandbox、rollout、thread runtime。

后续只跟踪会影响“请求能不能正确转发、响应能不能正确返回、日志计费能不能正确统计”的上游变化。

中转网关允许改写字段。`service_tier`、`model`、`reasoning`、`prompt_cache_key`、session 相关 header 这类字段被 CodexManager 按配置改写是正常行为，不应简单按“和上游原始请求不一致”判定为问题。

真正要检查的是：改写来源是否明确、覆盖优先级是否固定、最终出站值是否符合上游语义、日志是否能解释为什么这样转发。

## 3. 跟踪范围

### 中转改写原则

- 允许改写
  - API key profile 可以覆盖 `model`、`reasoning`、`service_tier`。
  - 模型转发规则可以把客户端模型改成上游模型。
  - compact 模型转发规则可以单独覆盖 compact 请求模型。
  - `service_tier` 可以按 CodexManager 配置或上游兼容规则映射，例如把兼容输入归一成上游可接受值。
  - `prompt_cache_key`、`session_id`、`x-client-request-id`、`x-codex-turn-state` 可以为了账号切换、线程锚点、failover、conversation binding 做亲和性处理。
  - 出站 `Authorization` 必须改成实际被选中账号或聚合 API 的 token。

- 不允许无解释改写
  - 用户或 API key 显式配置了 `service_tier`，但日志看不出为什么最终为空或变成其他值。
  - `balanced` 策略被 manual preferred account、conversation binding、prompt cache key binding 覆盖，但执行计划仍显示像是普通 balanced。
  - 模型转发后，上游模型、计费模型和日志模型三者无法对应。
  - 协议转换后丢失 tools、reasoning、usage，且日志仍标记为正常转发。

- 判断标准
  - 看最终出站值，不只看入站请求。
  - 看覆盖来源，不只看字段是否变化。
  - 看上游是否接受，不只看是否和官方 Codex 本体完全同源。
  - 看日志能否复盘，不只看当前请求是否成功。

  - 只要格式不影响请求，不需要强行一致。

  - 只跟踪会被服务端校验或影响兼容的字段。

- [x] 聚合 API 协议转换后丢失 reasoning / tool / usage。

## 7. 每次上游同步检查步骤

1. 更新上游 Codex 源码。
2. 查看上游最近提交是否触及 `client.rs`、`login`、`model-provider-info`、`codex-api`。
3. 如果只触及 `tui`、`app-server`、`cloud-config`、`docs`，默认不进入 CodexManager 实现。

5. 如果影响中转，先写最小回归测试，再改网关。
6. 验证至少覆盖一条真实链路：HTTP `/v1/responses`、streaming、WebSocket 或协议转换。
7. 更新本清单或对应差异报告。

## 9. 当前建议

当前不建议做 app-server / thread runtime 方向。

建议后续只开一个“Codex 上游中转兼容”维护项，按 P0 / P1 清单滚动检查。真正要动代码时，优先改协议字段、header、token、WebSocket、usage、日志和覆盖链可观测性，不扩大到 Codex agent 本体。

对 `service_tier` 这类字段，默认按“中转正常改写”处理。只有当最终出站值不符合上游语义、配置优先级错误、或日志无法解释覆盖来源时，才进入修复范围。

## 10. 第一版落地记录

本次先落地 `service_tier` 覆盖链可观测性，不改变请求改写行为。

- 新增请求日志字段 `service_tier_source`，随 `service_tier` 和 `effective_service_tier` 一起返回。
- 来源口径先按可验证链路归类：
  - `client_request`：客户端显式值就是最终生效值。
  - `api_key_profile`：客户端没有显式值，最终值来自 API key profile 的 `service_tier` 配置。
  - `gateway_config`：客户端没有显式值，最终值来自中转侧配置或默认注入。
  - `gateway_override`：客户端显式值被中转侧最终值覆盖。
  - `unset`：客户端和最终值都为空。
- 日志页模型/推理 tooltip 增加“服务等级来源”，方便排查为什么最终按该 service tier 出站。
- 请求日志搜索增加 `tier_source:` / `service_tier_source:` 前缀，便于筛选覆盖来源。
- 覆盖范围包括普通 HTTP 主链路、Responses WebSocket、Aggregate API 转发，以及模型路由 / 聚合路由错误日志路径。

后续如果要继续细拆来源，可以在请求准备阶段继续把 `account_policy`、`codex_backend_compat` 等更细配置来源传入 `service_tier_source`。

## 12. 第三版落地记录

本次继续推进 `balanced` / `ordered` 路由策略覆盖链可观测性，不改变候选选择和 failover 行为。

- 新增请求日志字段 `route_strategy`、`route_source`。
- `route_strategy` 记录实际用于日志解释的路由策略标签，例如 `ordered`、`balanced`、`conversation_bound`、`manual_preferred_account`。
- `route_source` 记录策略来源：
  - `route_strategy`：按全局 gateway route strategy 排序或轮询。
  - `manual_preferred_account`：手动优先账号把候选起点固定到指定账号。
  - `conversation_bound`：已有 conversation / prompt cache key binding 把账号固定到队首，覆盖普通 `balanced` 轮询。
  - `aggregate_api_preferred`：聚合 API 显式指定供应商优先，覆盖普通聚合 API balanced 轮询起点。
- 普通 HTTP account 路径复用真实 `CandidateRotationPlan` 写入日志，避免日志口径和实际候选顺序分叉。
- Aggregate API 路径记录当前 gateway route strategy，并在显式 `aggregate_api_id` 时记录 preferred 覆盖来源。
- Responses WebSocket 路径记录当前 gateway route strategy，保持 WS 日志和 HTTP 日志字段结构一致。
- 日志页“路径 / 路由”tooltip 增加“路由策略”和“路由来源”，用于解释为什么 balanced 看起来没有轮询。
- 请求日志搜索增加 `route_strategy:` / `strategy:`、`route_source:` / `route_reason:` 前缀。

该版本主要解决“balanced 被 manual preferred account、conversation binding、prompt cache key binding 覆盖但日志没有说清楚”的问题；仍不引入上游 Codex agent runtime。

## 13. 第四版落地记录

本次继续推进 Responses WebSocket 日志和 usage 采集与 HTTP streaming 路径的一致性，不改变 WebSocket 转发行为。

- WebSocket terminal detection 新增 `response.incomplete`，这类上游终止事件现在会按失败终止写入请求日志，避免请求只等连接关闭才落日志。
- `response.incomplete` 的错误信息优先读取 `response.status_details.error.message`；没有上游错误详情时，使用和 HTTP streaming 一致的“连接中断（可能是网络波动或客户端主动取消）”提示。
- WebSocket usage 解析补齐 Chat Completions 兼容字段：
  - `prompt_tokens_details.cached_tokens` 映射到 `cached_input_tokens`。
  - `completion_tokens_details.reasoning_tokens` 映射到 `reasoning_output_tokens`。
- 保持 `response.usage` 优先于顶层 `usage` 的口径，和 HTTP SSE / 非流式日志采集保持一致。
- 补充 WebSocket 单元测试覆盖兼容 usage 字段和 `response.incomplete` terminal 事件。

该版本主要解决“WS 上游返回兼容 usage 或 incomplete terminal 时，日志/计费采集和 HTTP 路径不一致”的问题；仍不改变账号选择、failover 和上游请求 shape。

## 14. 第五版落地记录

本次继续推进 Aggregate API 协议转换保真，重点补齐 Responses -> Anthropic Messages bridge 里的工具调用策略，不改变聚合 API 路由、认证、重试和响应桥接行为。

- Responses 请求中的 `parallel_tool_calls: false` 现在会在转 Anthropic `/v1/messages` 时映射为 `tool_choice.disable_parallel_tool_use: true`。
- 指定函数工具调用的 `tool_choice` 继续保留为 Anthropic 的 `{ "type": "tool", "name": ... }`，同时附带禁用并行工具标记。
- 保留既有 reasoning 转换行为：
  - Responses `reasoning.effort` 继续映射为 Anthropic `thinking`。
  - `reasoning.effort: none` 继续映射为 disabled thinking。
  - reasoning budget 继续保持低于 `max_tokens`。
- 保留既有 usage 采集行为：Aggregate API bridge 响应仍通过 `bridge.usage` 写入请求日志和 token stats。
- 补充 `responses_anthropic_bridge_preserves_tool_choice_parallel_policy` 回归测试，并复跑 `request_router` 和 `aggregate_api` 测试过滤。

该版本主要解决“Aggregate API Claude bridge 会丢失 Responses 工具并行策略”的问题；reasoning / usage 已由现有 bridge 测试继续覆盖，本次未扩大到 Codex agent runtime。

## 15. 第六版落地记录

本次继续推进上游 usage 结构兼容，重点补齐 Responses-from-Anthropic 流式桥接路径里的兼容字段采集，不改变请求转发、账号路由、failover 和响应事件转换行为。

- `ResponsesFromAnthropicSseReader` 的 usage 采集现在同时兼容 Anthropic 原生字段和 OpenAI / Responses 兼容字段：
  - `prompt_tokens` 映射为 `input_tokens`。
  - `prompt_tokens_details.cached_tokens` / `input_tokens_details.cached_tokens` 映射为 `cached_input_tokens`。
  - `completion_tokens` 映射为 `output_tokens`。
  - `completion_tokens_details.reasoning_tokens` / `output_tokens_details.reasoning_tokens` 映射为 `reasoning_output_tokens`。
  - `total_tokens` 存在时优先信任上游总数；不存在时保留原 Anthropic 兜底口径。
- 补充 `anthropic_to_responses_reader_accepts_openai_usage_fields` 回归测试，覆盖桥接流式响应最终 SSE payload 和请求日志 usage collector。
- 该项属于中转兼容范围：上游或聚合供应商如果在 Anthropic SSE 形态中返回 OpenAI-style usage，日志、计费和统计不应丢 cached / reasoning token。

该版本主要解决“上游 usage 结构变化导致桥接流式路径日志/计费字段丢失”的问题；仍不引入 Codex app-server、TUI、MCP、sandbox、thread runtime 等非中转模块。

## 16. 第七版落地记录

本次继续推进上游模型列表能力字段兼容，重点补齐 Codex `/models` 模型能力元数据，不改变模型路由、账号选择和请求转发行为。

## 18. 第九版落地记录

本次推进 P0 的 ChatGPT token refresh / unauthorized retry 对齐，重点补齐 Responses WebSocket 握手 401 后的刷新重试，不改变普通 HTTP 主链路、账号候选排序、failover 和请求 payload。

- 对照上游 Codex `core/src/client.rs`，当前上游 HTTP Responses 和 Responses WebSocket 在 401 时都会走一次 `refresh_token` recovery，然后用新认证状态重试。
- CodexManager 普通 HTTP upstream 已有 401 后 refresh-token 重试；本次补齐 WebSocket 握手路径：
  - WebSocket 握手错误现在保留 HTTP status code，不再只保留字符串错误。
  - 当前账号握手返回 401 时，如果不是 OpenAI API base 且账号有 `refresh_token`，会刷新并持久化 ChatGPT access token。
  - 刷新后会重新解析 WebSocket bearer，并用新 bearer 对同一账号重建握手请求重试一次。
  - 如果 refresh 失败，沿用现有 refresh-token 错误分类，把明确不可恢复的账号标记为 unavailable。
- 保持原 failover 行为：刷新不可用或刷新后仍失败时，继续按现有候选账号连接/failover 逻辑处理。
- 补充 `websocket_connect_error_preserves_http_unauthorized_status` 回归测试，防止 WebSocket 握手 401 再次退化成无法判断状态码的普通字符串错误。

该版本主要解决“上游 ChatGPT access/api-key bearer 过期时，HTTP 请求能 refresh retry，但 Responses WebSocket 握手 401 不能按上游行为刷新重试”的问题；仍不实现上游 app-server、thread runtime、TUI、MCP 或 sandbox。
